# Component and functionality catalogue

This is the code-backed catalogue for functionality owned by the PNS theme.
It is for developers deciding where a behavioural change belongs. It does not
define editorial layout recipes: use the template and pattern inventory, then
verify the live WordPress record before changing a route or composition.

## Ownership rule

The theme owns presentation, editor presentation, small site-specific bridges,
and the default data it seeds. Plugins own their content models, custom blocks,
and service integrations. WordPress owns saved content and Site Editor records.
See [ownership and integrations](../architecture/ownership-and-integrations.md)
for the full boundary map and the current required-plugin contract.

When removing a theme bridge, prove both that its upstream owner has replaced
the behaviour and that the affected frontend and editor surfaces still work.
Do not move a portable content-model or service concern into the theme merely
because the current templates use it.

## Change checklist

For a behavioural change, record the owning module and then:

1. Check whether a plugin or a saved WordPress record is the true owner.
2. Keep render filters narrow and retain their documented removal gate.
3. Run `pnpm check`, then the focused visual/editor route that exercises the
   change. Use the shop, navigation, template, or editor lane where applicable.
4. For a database-backed surface, capture the existing record and verify the
   deployed result rather than assuming a file changed the live site.
5. Update this catalogue together with any lasting new integration, dynamic
   block, editor control, bridge, or removal condition.
